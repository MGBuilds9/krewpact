Key Boundaries

  • GPL v3 Firewall — ERPNext accessed only via REST API through Cloudflare Tunnel. No shared code. Prevents license contamination.
  • RLS Boundary — Every Supabase query runs under Clerk JWT claims. Division + org scoping enforced at the database level.
  • BFF Pattern — API routes are thin orchestrators: authenticate, validate, call lib/, return. No business logic in routes.
  • Offline Boundary — IndexedDB queue syncs to Supabase. Conflict resolution per entity type (last-write-wins, merge, keep-both).

Connection Patterns

  • ERPNext: Token auth, 300 req/min, 3 retries, circuit breaker (opens after 5 failures, 30s reset)
  • Supabase: Pooler port 6543 (transaction mode) from serverless. Direct port 5432 never used.
  • QStash: HTTP POST with HMAC verification. Dead-letter after 3 failures. In-memory fallback for dev.
  • Clerk: Third-Party Auth. Native session token passed to Supabase. No custom JWT template.

Sync Rules

  • Outbox/inbox pattern with idempotent upsert
  • Retry with exponential backoff
  • Dead-letter queue after 3 failures
  • No two-phase commit — eventual consistency
  • ERPNext rate limit: 300 req/min
  • Inventory: ERPNext gets cost journal entries only

What Flows Where

  • KP → ERP: New customers, estimates approved → sales orders, timesheets → invoices, expenses → claims
  • ERP → KP: Invoice status, payment confirmations, GL snapshots, stock levels
  • Bidirectional: Project status, supplier updates, employee changes

Authentication & Authorization Flow

Clerk owns identity. Supabase enforces data isolation via RLS. The JWT carries org, division, and role claims that the database trusts.

1

User hits any route

proxy.ts (Next.js middleware) runs clerkMiddleware with auth.protect(). Public routes (/sign-in, /api/web/*, /api/health) are excluded. Everything else requires authentication.

2

Clerk issues session token

Third-Party Auth integration. Clerk's native session token (not a custom JWT template) contains metadata with custom claims set during user provisioning.

3

JWT claims extracted

Token carries: metadata.krewpact_user_id, metadata.krewpact_org_id, metadata.division_ids (JSONB array), metadata.role_keys (role array).

4

Supabase client created with token

createUserClient() passes the Clerk token as Authorization: Bearer header. Supabase validates the JWT and extracts claims for RLS evaluation.

5

RLS policies enforce isolation

Every table has RLS enabled (deny by default). Policies use auth.jwt() -> 'metadata' -> 'division_ids' to filter rows. The krewpact_divisions() function resolves division codes to UUIDs. Never auth.uid() — Clerk UUIDs differ from Supabase.

6

Role-based UI gating

FeatureGate component checks org_settings.feature_flags at the layout level. Admin bypass built in. 10 route layouts gated. Navigation items hidden when flag is absent.

Canonical Roles (13)

  • Internal (9): platform_admin, executive, operations_manager, project_manager, project_coordinator, estimator, field_supervisor, accounting, payroll_admin
  • External (4): client_owner, client_delegate, trade_partner_admin, trade_partner_user

Three Client Types

  • createUserClient() — RLS-scoped via Clerk JWT. Used in API routes for authenticated queries.
  • createUserClientSafe() — Same as above but returns {client, error} instead of throwing.
  • createServiceClient() — Bypasses RLS using service_role key. Server-only. Used for cron jobs, webhooks, admin operations.

Critical Rules

  • Never use auth.uid() in RLS — Clerk UUIDs != Supabase UUIDs
  • Never use service_role key in client code
  • Index every column referenced in RLS policies
  • Always validate Server Action input with Zod (TypeScript types are NOT runtime-enforced)
  • Division IDs in JWT are codes ["contracting","homes"] resolved to UUIDs by krewpact_divisions()
